Conception Page

Un descriptif du site est détaillé sur cette page.

Le présent site est composé de deux parties distinctes :

  • La partie Serveur hébergée sous Symfony 2.8 LTS dont le développement est consultable sur http://github.com/3kynox/portfolio-server.
  • La partie Client est une Single Page Application utilisant AngularJS dont le développement est consultable sur http://github.com/3kynox/portfolio-client.

Le Serveur

Le serveur utilise donc le Framework PHP Symfony et fournit une API-REST accessible par tous client autorisés.

Depuis Symfony 2.8, la possibilité d'utiliser le Framework avec un minimum de composants, nommé le Micro-Framework Symfony à été proposé. Le but bien évidemment est de gagner en vitesse & réactivité.

Le premier commit du portfolio-server disponible sur GitHub fait suite à un travail important effectué sur ce Micro-Framework lors de mon dernier passage chez Infonova ou des tests poussés et une multitude de configurations différentes ont pu être éprouvées afin d'obtenir les meilleurs performances.

Couplé à PHP7, on obtient des vitesse d'affichage des données de quelques milli-secondes.

Bundles Utilisés

- API-Platform : Framework permettant de mettre en place les fonctions API-Rest avec la mise à disposition du JSON-LD (JavaScript Object Notation for Linked Data) et du vocabulaire Hydra-Core qui sont tout simplement de nouveaux standards en matière d'API.

- Nelmio CORS Bundle : Gère les erreurs liées au Cross-Domain en javascript.

- Lexik JWT Bundle : Pas encore mis en place mais le sera prochainement. Se charge de sécuriser les échanges de données entre le client et le serveur.

- FOS User Bundle : Pas encore mis en place également. Gère l'identification utilisateurs avec Symfony avec quelques fonctionnalités poussées.


Le Client

- SPA (Single Page Application)

- Entièrement codé avec HTML5 / JS / AngularJS

- Étant donné que sur une SPA, le code devient très rapidement "fouilli", une volonté de séparer les couches à été mis en place. Cela se passe déjà au niveau de l'URL :



Normalement, sur la page d'index ne se trouve que les scripts de base à charger, incluant angular.

La balise sur index.html (relié au module de routing d'angular) renvoie vers le module "main" pour traiter le menu latéral et la topbar.

A l'intérieur de la vue html du module "main" se trouve une autre balise qui elle se charge de charger un autre module (dashboard, conception, ou autre ...) qui correspond à n'importe quelle page disponible du site.

On se retrouve avec une arborescence composé d'une page index.html avec un dossier "modules" composé du module "main" et de chaque module du site correspondant à chaque page du site. Chaque module est composé lui-même d'un fichier applicatif, d'un fichier définisant la route ainsi que d'un dossier "controllers", "views", "plugins" et "assets" pour gérer chaque aspects du module en question.

On obtient grâce à cette structure une SPA propre, bien découpée, et compatible (avec la partie serveur) avec un modèle Model View Controller.

- Utilisation du plugin ocLazyLoading : En complément, le lazyLoad permet de ne charger que les scripts ou css spécifique à chaque module! Ce qui veut dire que le module "main" possède des scripts globaux, et chaque module de page du site peut executer uniquement ses propres scripts. On voit couramment sur un site développé avec angular une pléthore de scripts exécutés tout au long de la navigation de l'internaute en permanence. Ici le problème est résolu.

- Utilisation du plugin RestAngular qui lui se charge d'aller chercher les données du serveur Symfony.



Un bug est actuellement présent avec le module "Main" et reste à corriger, ce qui explique pourquoi le menu et la topbar se trouve dans le index.html.
